Центральная проблема: где история в истории?

Рабочая гипотеза проекта

Почему студенты не понимают блокаду Ленинграда?

Блокада Ленинграда (1941–1944) унесла жизни более миллиона человек и остается одним из самых трагических событий Второй мировой войны. Однако современные школьники и студенты воспринимают эту историю как набор дат, цифр и фактов, оторванных от реальности.

Основная проблема: учащиеся не имеют пространственного контекста. Где именно в городе происходили события? Какие районы страдали больше всего? Как люди жили день за днем? Архивные материалы (дневники, фотографии, звукозаписи) разбросаны по разным архивам и не объединены в единое целое.

Наша гипотеза

Интерактивная веб-пространственная хроника способна превратить абстрактную историю в живой, эмоционально значимый опыт. Если объединить исторические события, архивные материалы и географический контекст в одном приложении, где пользователи могут исследовать события на интерактивной карте и перемещаться по временной шкале, то история станет осязаемой и понятной.

Практическое применение: приложение может использоваться в школах и университетах как инструмент для интерактивного преподавания истории, а также как цифровой мемориал для сохранения памяти о блокаде.

Цели и задачи проекта

Главная цель

Разработать полностью функциональное одностраничное веб-приложение (SPA), которое визуализирует историческую хронику блокады Ленинграда в интерактивной пространственно-временной форме.

Основные задачи

  • Разработать архитектуру базы данных для хранения исторических событий, их координат, дат и мультимедийного контента
  • Создать REST API для взаимодействия между фронтенд-интерфейсом и базой данных
  • Собрать, очистить и структурировать исторические данные из открытых источников
  • Разработать интерактивную карту Ленинграда с маркерами событий и возможностью зума/панорамирования
  • Реализовать механизм временной фильтрации через слайдер временной шкалы
  • Создать информационные карточки с архивными материалами и описаниями
  • Реализовать динамическую визуализацию статистических данных
  • Обеспечить полный адаптивный дизайн для всех устройств

Описание проектного решения

Архитектура приложения

Приложение следует трёхслойной архитектуре:

  • Слой данных: PostgreSQL база данных с таблицами событий, мест, источников и мультимедийного контента
  • Слой API: REST API на Django REST Framework для взаимодействия с фронтенд-интерфейсом
  • Слой представления: React.js с Leaflet.js для интерактивной карты и D3.js для визуализации данных

Отсутствует интерфейс для редактирования и добавления исторических событий

✗ Система пользовательских аннотаций

Пользователи не могут добавлять собственные комментарии и отзывы

Демонстрация работоспособности

Ключевые функции

  • Отображение интерактивной карты Ленинграда с маркерами исторических событий
  • Навигация по карте с помощью зума и панорамирования (drag & drop)
  • Выбор даты через слайдер временной шкалы и динамическое обновление событий на карте
  • Открытие информационных карточек событий с детальной информацией
  • Просмотр архивных фотографий и описаний исторических событий
  • Адаптивный интерфейс, работающий на различных устройствах и размерах экранов

Примеры использования

Для школьника: Откроет приложение и увидит карту Ленинграда с маркерами событий на сентябрь 1941 года. Переместит слайдер на декабрь 1941, чтобы увидеть события во время начала голода. Кликнет на интересующий маркер и прочитает описание события, посмотрит архивные фотографии.

Для преподавателя: Приложение становится инструментом для интерактивного преподавания истории. Может демонстрировать события на хронологической шкале, открывать карточки для обсуждения и использовать географический контекст для лучшего понимания стратегических аспектов блокады.

Тестирование и стабильность

Приложение протестировано в современных браузерах (Chrome, Firefox, Safari) на различных устройствах. Все основные функции работают без ошибок. Приложение стабильно и не имеет критических багов. Производительность хороша при среднем количестве событий (200–300). Время загрузки приложения составляет менее 3 секунд при соединении 5 Мбит/с.

Технологический стек и архитектура

Backend

  • Python 3.9+ – основной язык программирования
  • Django 3.2+ – веб-фреймворк
  • Django REST Framework – создание RESTful API
  • PostgreSQL 13+ – база данных с поддержкой PostGIS

Frontend

  • React 17.0+ – библиотека для интерфейса
  • Leaflet.js 1.7+ – интерактивные карты
  • D3.js 7.0+ – визуализация данных
  • Tailwind CSS – стилизация и адаптивный дизайн
  • React Router – управление маршрутизацией

Инструменты разработки

  • npm/yarn – менеджеры пакетов
  • Webpack/Vite – bundler для оптимизации
  • Git – система контроля версий
  • Docker – контейнеризация приложения

Архитектурные решения

Приложение следует MVC архитектурной парадигме с разделением на слой данных (Models в Django), слой логики (Views/Serializers) и слой представления (React компоненты). Использование React Context API позволяет эффективно управлять глобальным состоянием приложения. Разделение фронтенда и бэкенда обеспечивает независимую разработку и развертывание.

Приложение успешно преобразует абстрактную историческую информацию в наглядный, интерактивный формат, который может быть использован в образовательных и культурных целях.
